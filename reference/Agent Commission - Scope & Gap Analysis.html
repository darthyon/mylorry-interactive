01

At a glance

Agent Commission calculates, tracks and surfaces fuel-rebate earnings for internal sales Agents (~7–10), based on the fuel volume of the SP Accounts they manage. It is fuel-only — it does not touch MyAdmin / MyTrip subscription revenue. After the PRD strikethroughs, V1 is a calculation & visibility system; money movement (payouts) stays manual/offline.

The two-layer model (authoritative)
Commission = Σ per org [ Org Monthly Litres × Volume Tier Rate ] × KPI Multiplier %

Layer 1 — Base rate: each org's monthly volume maps to the agent's own volume-tier table (RM/L). Each org can land in a different tier. Layer 2 — KPI multiplier: the agent's total volume across all orgs vs their KPI target → 100% / 50% / 0%. Tier table is set in Agent Commission Config, not the SP Account Rebate Setting.

02

Source reconciliation

  • The originally-linked doc bundles three PRDs (Subscription · Subsidy Quota · Commission & KPI). This analysis uses the standalone Agent Commission Module PRD v1.2 as the source of truth.
  • Model differs from the earlier BRD. v1.2 is a per-org volume-tier rate × KPI multiplier model (multiple configurable tiers per agent), not a single flat base-rate × KPI%. We design to v1.2.
  • Org grouping (Master/Member) affects org rebate only — never agent commission. Commission is always calculated per-org individually (PRD §4G). Agent commission views must not show pooled grouping.
  • Subscription PRD §10 confirms Agent Commission is an independent workstream — not blocked by, and not blocking, Subscription go-live.

06

Discrepancies & decisions

Where the strikethroughs left dangling references, or the PRD contradicts itself, here's the design call taken this round.

DiscrepancyStatusDecision
FR-HM-01 header shows "Total pending payout requests" — but payout is parkedDecidedReplace with "Orgs within 60 days of validity expiry" (ties to FR-CV-07), or total period litres.
FR-HM-02 / 03 carry a "Payout status" column & filterDecidedReplace with a period state: Provisional → Finalized (after the 4th-of-month rebate run). Useful, not payout-coupled.
FR-AG-04 history "status (Pending / Approved / Paid)" are payout statusesDecidedSame substitution — Provisional / Finalized, not payout statuses.
US-HC-07 struck but FR-HM-07 (Host 12-mo history) not struckDecidedKeep the history display — FR-HM-07 stands; it's read-only audit, low cost. (Flagged for your confirmation.)
New Org exception self-contradicts on mid-year starters (Org D/E vs Org F)DecidedBadge driven by a data flag (Auto 100% / Custom % / —), never a UI date rule. PRD §4D confirms three modes.
Existing Figma "Commission / Tier" fields live under SP Account → RebateDecidedThose are org rebate, a different thing. Agent volume-tier table lives in Per-Agent Config (PRD §4, §11).
Petron 2–3 hr sync; commission on settled txns only (R7, FR-HC-10)DecidedShow a "last synced" timestamp in commission views; current month is Provisional until settled.

Open questions — need an answer before/at build

  • Confirm §06 decisions — especially Provisional/Finalized replacing payout statuses, and keeping the Host 12-mo history (FR-HM-07).
  • OQ4 — payout cycle (PRD open): moot for V1 visuals since payout is parked, but confirms there is no "request" affordance in the Agent dashboard.
  • OQ5 — reversals/refunds deducted from commission base? Affects how the SP breakdown reconciles. Owner: Finance/Eng
  • OQ6 — gross vs net (post-subsidy) litres as the commission base? Affects the volume figure shown everywhere. Owner: Finance
  • OQ10 — approval/paid notifications (low; payout-adjacent, likely parked with payout).
  • Termination & transfer UI — the 4th-of-month cutoff + "hold state" (FR-TM-06, FR-TR-06) is intricate; needs an agreed visual model on the Config page.
08

Risks & dependencies that touch design

  • R2 — Bug #6 (commission grouping) is the trigger for this work and still open. Our one-row-per-org-per-month statement structure is the design-side mitigation.
  • R7 — Petron sync delay. Drives the "last synced" timestamp + Provisional/Finalized states.
  • Agent auth scoping (PRD §3 ⚠): an agent must only ever see their own data — the Agent Portal sits outside the Host/Org taxonomy. Design must avoid any cross-agent surface.
  • R4 — no agent bank details collected. Was a payout prerequisite; with payout parked, it's no longer a V1 blocker — noted.
